# SPEC — habr-corpus-parser

Парсер статей с habr.com в markdown-корпус. Python 3.12.

## 1. Цель

Собирать статьи с Хабра в локальный markdown-корпус, который служит одновременно:

1. **Базой памяти о собственных публикациях** — чтобы помнить, о чём уже писал, строить отсылки и серийность между статьями. Это ядро проекта.
2. **Референсной библиотекой** — чужие статьи по интересующим темам/авторам для опоры и вдохновения.
3. **Корпусом для NLP / лингвистики и ML** — чистый текст с метаданными для анализа языка, тем, трендов и обучения моделей.
4. **Читаемым архивом** — статьи сохраняются в удобочитаемом виде (вёрстка, код, картинки).

Эти цели диктуют главный приоритет: **чистое тело статьи** (для анализа) + **богатые метаданные** (для поиска, отсылок и статистики).

## 2. Что парсим

Источник статей задаётся двумя способами:

- **По автору / пользователю** — все публикации указанного автора.
- **По поиску / фильтрам** — ключевые слова, теги/хабы, даты (как в поиске Хабра).

### Разделение «свои / чужие»

- В конфиге указывается **свой username** на Хабре.
- Статьи этого автора → `corpus/mine/`.
- Все остальные → `corpus/reference/`.
- Авторство определяется по полю автора из данных статьи, а не по способу запуска.

### Типы контента

Включаем в корпус:

- **Переводы** — включаем, помечаем в мете `translation: true` (и по возможности `original_url`).
- **Корпоративные блоги** — включаем; в мете фиксируем компанию.
- **Ньюсы / мегапосты** — включаем.

Фильтра по минимальной длине нет (лонгриды и короткие посты идут наравне). Возможность добавить порог длины — на будущее, не в первой версии.

## 3. Объём

На старте объём не определён — начинаем с малого, но архитектура должна выдержать рост до тысяч статей: устойчивость к сбоям, докачка, инкрементальное обновление. Хранилище — плоские файлы (без БД); при выходе на десятки тысяч вопрос БД пересматривается отдельно.

## 4. Способ получения данных — гибрид API + HTML

Хабр отдаёт данные через неофициальный JSON API (тот, что использует мобильное приложение) и через обычный HTML. Используем **гибрид**:

- **JSON API — приоритетно** для: списков статей (по автору, по поиску), метаданных, метрик, тела статьи (приходит как HTML-фрагмент), комментариев. Даёт чистые структурированные данные и меньше возни с вёрсткой.
- **HTML — фолбэк** когда API недоступен, вернул неполные данные или изменил формат. Парсим страницу статьи напрямую.

Конкретные endpoint'ы API уточняются на этапе реализации (API неофициальный и может меняться) — это инкапсулируется в отдельном слое-клиенте, чтобы смена API не задевала остальной код.

> Тело статьи в обоих случаях приходит как HTML → конвертируется в markdown (см. §7).

### Вежливость к habr.com (обязательно, из CLAUDE.md)

- Задержка **2–3 секунды** между запросами.
- Внятный **User-Agent** с контактом.
- Ретраи с экспоненциальным бэкоффом на сетевых ошибках / 429 / 5xx.
- **Без авторизации** — только публичные данные. Закрытые статьи и метрики, требующие логина (точные просмотры, закладки), не собираем; недоступные поля остаются пустыми.

## 5. Структура корпуса

```
corpus/
  mine/
    2024-03-15-kak-ya-parsil-habr.md
    2024-03-15-kak-ya-parsil-habr.comments.md
    assets/                      # только при --download-images
      2024-03-15-kak-ya-parsil-habr/
        img-01.png
  reference/
    2023-11-02-ivanov-pro-rag.md
    2023-11-02-ivanov-pro-rag.comments.md
  INDEX.md                       # сводный человекочитаемый индекс
  corpus.jsonl                   # вся мета построчно, для программной обработки/RAG
state.json                       # состояние докачки/прогресса
errors.log                       # лог сбоев
```

- Внутри `mine/` и `reference/` — плоско.
- **Имя файла:** `{дата-публикации}-{slug}.md`, где `slug` — из URL статьи (транслит-хвост). Дата спереди даёт естественную хронологическую сортировку (важно для серийности). При коллизии добавляется `-{id}`.
- **Комментарии** — в отдельном файле `{тот-же-basename}.comments.md`, чтобы тело статьи оставалось чистым. (Фильтр «только топ-комментарии по рейтингу» — опция на будущее.)

## 6. Формат markdown-файла статьи

Frontmatter (YAML) + тело.

```markdown
---
# — базовые —
id: 123456
url: https://habr.com/ru/articles/123456/
title: Как я парсил Хабр
author: vitalijturov
author_url: https://habr.com/ru/users/vitalijturov/
published: 2024-03-15T10:30:00+03:00
# — классификация —
hubs: [python, data_mining]
tags: [парсинг, habr, nlp]
company: null                    # для корпоративных блогов
type: article                    # article | news | megapost
translation: false
original_url: null               # для переводов
# — метрики (public, могут быть неполными без авторизации) —
rating: 42
views: 15000
comments_count: 37
bookmarks: null                  # недоступно без авторизации
# — производные —
word_count: 2380
reading_time_min: 12
summary: null                    # опционально, генерация на будущее
# — служебное —
source: api                      # api | html — откуда взято тело
parsed_at: 2026-07-04T12:00:00+03:00
modified: 2024-03-16T09:00:00+03:00   # дата последнего изменения статьи на Хабре
---

# Как я парсил Хабр

<тело статьи в markdown: заголовки, списки, цитаты сохраняются>

​```python
# блоки кода — с сохранением языка
print("hello")
​```

Формулы сохраняются (LaTeX/inline по возможности).

![alt-текст](https://habrastorage.org/....png)   # ссылка по умолчанию
```

- **Картинки** — по умолчанию оставляем ссылки + `alt`. Флаг `--download-images` скачивает их локально в `assets/{basename}/` и переписывает пути в файле.
- **Код** — блоки с сохранением языка. **Формулы** — сохраняем разметку по возможности.

### Файл комментариев `{basename}.comments.md`

```markdown
---
article_id: 123456
article_url: https://habr.com/ru/articles/123456/
comments_count: 37
parsed_at: 2026-07-04T12:00:00+03:00
---

- **user1** (+15, 2024-03-15T11:00): Текст комментария…
  - **user2** (+3, 2024-03-15T11:20): Ответ (сохраняем вложенность).
```

## 7. Конвертация HTML → markdown

- Тело статьи (HTML-фрагмент из API или из страницы) чистится и конвертируется в markdown.
- Сохраняем: заголовки, списки, цитаты, таблицы, ссылки, блоки кода (с языком), формулы, картинки (alt + src).
- Вырезаем: рекламные вставки, «читайте также», кнопки шеринга, служебную обвязку.

## 8. Инкрементальное обновление (умное)

При повторном запуске для уже скачанной статьи:

1. Сравниваем дату изменения статьи на Хабре (`modified`) с сохранённой.
2. **Если статья редактировалась** → перекачиваем тело целиком.
3. **Если не редактировалась** → тело не трогаем, обновляем только метрики (`rating`, `views`, `comments_count`) и, при необходимости, файл комментариев.
4. Флаг `--force` игнорирует проверку и перекачивает всё заново.

## 9. Надёжность и обработка сбоев

- **Лог ошибок + продолжение:** сбой на одной статье не роняет весь прогон; проблема пишется в `errors.log` (URL, тип ошибки, время).
- **Ретраи с бэкоффом:** на сетевых ошибках, 429, 5xx — несколько повторов с растущей паузой.
- **Сохранение прогресса:** состояние в `state.json` — прогон можно прервать и продолжить с того же места (обработанные/оставшиеся URL).
- **Итоговый отчёт:** по завершении — сводка: скачано N, обновлено K, ошибок M, список проблемных URL.

## 10. Интерфейс — CLI + конфиг

Основные операции — через CLI; частые наборы (свои авторы, постоянные теги) — в конфиг-файле.

### Примеры команд

```bash
# все статьи автора (свой username из конфига → mine/, иначе reference/)
habr-parse author vitalijturov

# поиск по тегу/ключевым словам
habr-parse search --tag python --query "rag" --after 2023-01-01

# запуск набора из конфига
habr-parse run --config habr.yaml

# перекачать всё заново
habr-parse author vitalijturov --force

# скачивать картинки локально
habr-parse author vitalijturov --download-images

# пересобрать INDEX.md и corpus.jsonl из имеющихся файлов
habr-parse reindex
```

### Конфиг (`habr.yaml`)

```yaml
me: vitalijturov              # свой username → статьи идут в mine/
output_dir: corpus
delay_seconds: 2.5            # пауза между запросами (2–3 с)
user_agent: "habr-corpus-parser (contact: inforobot.vit@gmail.com)"
download_images: false
include_comments: true

targets:
  - type: author
    value: vitalijturov
  - type: author
    value: ivanov
  - type: search
    tags: [python, nlp]
    query: "rag"
    after: 2023-01-01
```

## 11. Сводный индекс

Генерируется/обновляется после парсинга (и по команде `reindex`):

- **`corpus/INDEX.md`** — человекочитаемая таблица всех статей: дата, заголовок, автор (свой/референс), хабы/теги, рейтинг, ссылка. Быстрый ответ на «о чём я уже писал».
- **`corpus/corpus.jsonl`** — по одной JSON-строке на статью со всей метой (без тела или с усечённым телом) для программной обработки, RAG и статистики.

## 13. Edge cases

- **Статья удалена / 404** → лог, пропуск.
- **Доступна только авторизованным** → пропуск (без авторизации), запись в лог.
- **Изменилась вёрстка / формат API** → фолбэк HTML↔API; при полной неудаче — лог и пропуск, прогон продолжается.
- **Пустое тело / только оглавление** → сохраняем с пометкой, `word_count: 0`.
- **Многостраничные / очень длинные статьи** → собираем целиком в один файл.
- **Коллизия имён файлов** → к basename добавляется `-{id}`.
- **Битые/относительные ссылки на картинки** → нормализуем в абсолютные; при `--download-images` — недоступные пропускаем с логом, ссылку оставляем.
- **Автор сменил username** → статьи привязываем по автору из данных статьи; смена `me` в конфиге влияет только на новые прогоны.
- **Rate limit (429) / временный бан** → бэкофф и ретраи; при устойчивом бане — остановка с сохранением прогресса.
